# NeuroAI Pipeline JSON Output Structures

This document describes the JSON schemas generated across different phases of the audio processing pipeline (Transcription, Diarization, Alignment, Stitching, Flagging, and Human Editing). Understanding these structures is crucial for debugging, frontend integration, and model evaluation/comparison workflows.

---

## 1. Core AI Outputs (Intermediate Phase)

These files are typically generated and stored in the `intermediate/` directory for each chunk or the entire case.

### `*_whisper.json`
* **Phase:** ASR (Whisper)
* **Description:** Raw transcription output from the Whisper model.
* **Structure:** Array of dictionaries with exact word/segment timestamps.
```json
[
  {
    "start": 0.52,
    "end": 1.84,
    "text": "今天天氣很好"
  },
  {
    "start": 1.86,
    "end": 3.10,
    "text": "我們出去玩吧"
  }
]
```

### `*_diar.json`
* **Phase:** Speaker Diarization (Pyannote / BiLSTM)
* **Description:** Continuous timeline of speaker identities.
* **Structure:** Array of dictionaries.
```json
[
  {
    "start": 0.50,
    "end": 1.90,
    "speaker": "SPEAKER_00"
  },
  {
    "start": 1.95,
    "end": 4.00,
    "speaker": "SPEAKER_01"
  }
]
```

### `*_aligned.json`
* **Phase:** Alignment
* **Description:** The intersection of Whisper text and Diarization speakers. Short segments are assigned a single predominant speaker.
* **Structure:** Array of dictionaries adding a unique `id`, `speaker` string, and basic `flag`.
```json
[
  {
    "id": "chunk_0_0",
    "start": 0.52,
    "end": 1.84,
    "speaker": "SPEAKER_00",
    "text": "今天天氣很好",
    "flag": "auto" 
  }
]
```
*(Note: If the `best_speaker` is "Unknown", the `flag` may be set to "review_needed".)*

## 2. Agent Processing & Optimization Outputs

After intermediate AI outputs are generated, several processing rules and LLM agents refine the dataset into a comprehensive transcript.

### `*_stitched.json`
* **Phase:** Rule-based or AI Stitching
* **Description:** Fragments from `*_aligned.json` are grouped by speaker and short pauses are stitched together to form coherent, readable sentences.
* **Structure:** Array of dictionaries similar to `_aligned.json` but with combined `text` and spanning `start`/`end` times.

### `*_verified_dataset.json`
* **Phase:** LLM Agent Stitching / Verification
* **Description:** High-quality merged text created by an LLM acting as a Data Archivist. Includes hallucination detection metadata.
* **Structure:** Array of dictionaries.
```json
[
  {
    "start": 0.52,
    "end": 3.10,
    "speaker": "SPEAKER_00",
    "text": "今天天氣很好，我們出去玩吧。",
    "source_ids": [0, 1],
    "verification_score": 0.95,
    "status": "Verified" 
  }
]
```
*(Note: `status` can be "Verified" or "Rejected_Raw_Kept" depending on the `verification_score`.)*

### `*_flagged_for_human.json`
* **Phase:** LLM Anomaly Detection (Flag Agent)
* **Description:** Checks ASR errors, context mismatches, or unintelligible speech. It respects repetitions (e.g., autism traits) without flagging them as errors.
* **Structure:** Array of dictionaries appending review keys.
```json
[
  {
    "sentence_id": 5,
    "start": 10.20,
    "end": 12.50,
    "speaker": "SPEAKER_01",
    "text": "天氣很難",
    "needs_review": true,
    "review_reason": "[Likely_ASR_Error] 讀音相近，應為天氣很藍",
    "suggested_correction": "天氣很藍"
  }
]
```

## 3. Final Outputs & Frontend Format

These files serve as the definitive "source of truth" and represent the overall status of a transcription project.

### `transcript.json` / `final_transcript.json` / `*_edited.json`
* **Phase:** Output Aggregation & Human Verification
* **Description:** The root-level complete document. `transcript.json` is generated by the pipeline. `*_edited.json` is saved by human annotators through the frontend.
* **Structure:** Contains global metadata, speaker mappings, and the flattened segment array.
```json
{
  "case_name": "20260513-1200-video",
  "video_file": "video.mp4",
  "processed_at": "2026-05-13T12:00:00.000000",
  "total_segments": 150,
  "flagged_segments": 3,
  "speaker_mapping": {
    "SPEAKER_00": "Teacher",
    "SPEAKER_01": "Student"
  },
  "segments": [
    {
      "sentence_id": 0,
      "start": 0.52,
      "end": 3.10,
      "speaker": "SPEAKER_00",
      "text": "今天天氣很好，我們出去玩吧。",
      "verification_score": 0.95,
      "status": "reviewed",
      "needs_review": false,
      "review_reason": null,
      "suggested_correction": null
    },
    // ...
  ]
}
```

## 4. Evaluation and Comparison

When evaluating pipeline performance (e.g., running scripts like `scripts/evaluate/audit.py` or comparing LLM configurations), the critical files are:

1. **Prediction (Hypothesis):** Usually `output/transcript.json` (or `*_stitched.json`/`*_verified_dataset.json` for specific component testing).
2. **Ground Truth (Reference):** `output/*_edited.json` (the exact file finalized by human editors).

The scripts flatten the text out of the `.segments` array and compute **Word Error Rate (WER)** and string similarity scores. The metadata (like `verification_score` and `flagged_segments`) is often used in EDA to correlate system confidence with real human correction rates.
